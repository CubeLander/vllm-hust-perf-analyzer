# TraceLoom Open-Source Refactor Plan

TraceLoom is already shaped as a standalone repository: it has packaging
metadata, documentation, examples, reproduction scripts, and a package entry
point. The remaining issue is that the core analyzer is still a transitional
copy of the research prototype. Before publishing it as an open-source project,
the implementation should be split along stable boundaries.

## Release Positioning

TraceLoom should be published as a lightweight offline analyzer, not as a full
profiling platform.

It should promise:

- read existing profiler outputs generated by vendor tools;
- build semantic anchor timelines;
- discover repeated execution structure;
- attach auxiliary/prelude costs;
- export CSV, JSON, Markdown, and eventually Perfetto-compatible timeline
  artifacts.

It should not promise:

- launching production workloads;
- installing accelerator drivers or vendor stacks;
- replacing Perfetto, Chrome Trace Viewer, Nsight, or `msprof`;
- fully automatic source attribution without human review.

## Current State

The public-facing repository layout is reasonable:

- `README.md`: project overview and usage;
- `pyproject.toml`: installable package and CLI entry points;
- `docs/`: architecture, input profile, output schema, reproduction notes;
- `examples/`: minimal configs and workload sketches;
- `reproduce/`: paper reproduction scripts;
- `traceloom/`: Python package.

The main technical debt is inside the package:

- `traceloom/compute_prelude_timeline.py` is about 3.5k lines and still mixes
  profile reading, event classification, symbolization, grammar discovery,
  metrics, rendering, CLI parsing, and output writing.
- `traceloom/loop_tree.py` is about 1k lines and contains both reusable grammar
  utilities and old msprof loading helpers.
- there are no committed tests or small synthetic fixtures.

## Minimal Refactor Before Public Release

The goal is not to redesign the algorithm. The goal is to preserve current
behavior while making the code reviewable, testable, and credible as an
open-source project.

Recommended module split:

| Target module | Responsibility |
| --- | --- |
| `traceloom/cli.py` | Public CLI argument parsing and command dispatch. |
| `traceloom/config.py` | Analyzer configuration dataclasses and defaults. |
| `traceloom/schema.py` | Shared dataclasses such as `StreamEvent`, `MainEvent`, device selections, and schema version constants. |
| `traceloom/readers/ascend_msprof.py` | Ascend/CANN SQLite loading, string-id resolution, table existence checks, and schema diagnostics. |
| `traceloom/anchors.py` | Semantic anchor classification, collective detection, kernel role overrides, and anchor timeline construction. |
| `traceloom/symbols.py` | Label normalization, symbol assignment, and symbol table construction. |
| `traceloom/grammar.py` | Pair-grammar macro discovery and adjacent macro-run loop promotion. |
| `traceloom/tree.py` | Tree payload construction, macro inlining, and readable tree rendering. |
| `traceloom/metrics.py` | Prelude/aux attribution, node cost metrics, macro metrics, and quality summaries. |
| `traceloom/exporters/csv.py` | CSV writers and stable output column ordering. |
| `traceloom/exporters/markdown.py` | Markdown report rendering. |
| `traceloom/exporters/perfetto.py` | Future augmented Perfetto/Chrome Trace JSON exporter. |
| `traceloom/pipeline.py` | Orchestration equivalent to the current `run_compute_prelude_timeline`. |

Keep compatibility during refactor:

- keep `traceloom.compute_prelude_timeline:main` as a compatibility wrapper;
- keep current output filenames unchanged for the paper artifact;
- introduce an explicit `schema_version` in `meta.json`;
- move one logical block at a time and compare generated outputs on the same
  sample profile.

## Test Strategy

Before public release, add small tests that do not require Ascend hardware:

1. synthetic event sequences for grammar and repeat detection;
2. synthetic aux/prelude interval attribution;
3. SQLite fixture with minimal `TASK`, `STRING_IDS`, and optional
   `COMMUNICATION_OP` tables;
4. CLI smoke test that writes an output bundle to a temporary directory;
5. golden-file checks for stable CSV headers and tree JSON fields.

The tests should validate the public contract, not every research heuristic.

## Paper Integration

The thesis should not contain a detailed system implementation chapter unless
it is required by the degree format. The paper's main contribution is the
structured profile representation and the trace-to-source attribution method.

Recommended thesis treatment:

- keep method details in the method chapter;
- mention TraceLoom briefly in the experiment setup as the prototype used to
  generate reports;
- put reproducibility details and raw profile artifact structure in the
  appendix;
- let the open-source repository carry package layout, CLI usage, and
  reproduction scripts.

This makes the thesis read like a method-and-evidence paper, while the
open-source repository carries engineering credibility.

## Suggested Release Checklist

- [ ] Split CLI/config/schema from the large pipeline file.
- [ ] Move Ascend SQLite loading into `readers/ascend_msprof.py`.
- [ ] Move grammar discovery into `grammar.py` with synthetic tests.
- [ ] Move metrics and exporters out of the pipeline.
- [ ] Add small fixtures and CI-ready tests.
- [ ] Freeze `schema_version = "0.1"` for output artifacts.
- [ ] Remove research-only paths and machine-specific assumptions.
- [ ] Confirm no large profiles, private paths, or benchmark logs are tracked.
- [ ] Tag the first public release as alpha.
