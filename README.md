# TraceLoom

This analyzer is named **TraceLoom**. The repository may be hosted as
`vllm-hust-perf-analyzer`, but the command-line tool, Python package, and
project identity remain `traceloom` / TraceLoom.

TraceLoom is a lightweight offline analyzer for accelerator profiler timelines.
It is designed for distributed deep-learning performance work: take a profile
generated by the user's native CUDA/Nsight or Ascend/CANN stack, recover
semantic loop and communication structure, then emit tables, reports, and cost
summaries.

TraceLoom is intentionally not a runtime platform. It does not ship a Docker
environment, install accelerator drivers, or provide an Ascend hardware stack.
The CLI can optionally invoke `msprof` from an editable config, but the analyzer
core remains offline and consumes profiler output after the workload has run.

## Status

Current implementation:

- Ascend/CANN `msprof` SQLite profile discovery and analysis;
- semantic anchor timelines for compute and collective events;
- loop/repeat tree construction over semantic anchors;
- aux/prelude attribution for events that prepare the following anchor;
- CSV, JSON, Markdown, and aggregate loop-cost outputs.

Planned public surface:

- CUDA/Nsight input adapter;
- reference reproduction scripts for paper results;
- stable output schema and quality reports.

## Analyze An Ascend Profile

The recommended user flow is config-driven and keeps setup, profiling, and
analysis loosely coupled:

```bash
traceloom create config -o traceloom.profile.ini
# edit workload, profiler, Docker, and output paths
traceloom run traceloom.profile.ini
traceloom analysis runs/local-msprof/msprof_raw --out-dir runs/local-msprof/analysis
```

`traceloom run` invokes Ascend/CANN `msprof` as:

```text
msprof --output=<profile_dir> --application=<workload.command> <profiler.extra_args>
```

If `[docker] enabled = true`, the profiler command is run through `docker exec`
when `container` is set, or through `docker run --rm` when `image` is set.

The analysis command consumes an existing `msprof` output directory. It does not
read the workload command from the config and does not launch the workload.
By default it analyzes every discovered device. To limit the scope, set
`[analysis] top_devices_global` or `[analysis] devices = 3,4,5,6` in the
profile config, or pass `--top-devices-global` / `--devices` on the CLI. The
template keeps bounded event and macro limits for interactive use; set
`max_main_events_per_device = 0` or `max_macro_defs = 0` only when you want an
exhaustive offline pass.

For direct analysis, TraceLoom still accepts either a run directory containing
`msprof_raw` or the raw profile directory directly:

```text
<run_dir>/msprof_raw/PROF_*/msprof_*.db
<raw_dir>/PROF_*/msprof_*.db
```

Run:

```bash
traceloom analysis <run_dir-or-raw-msprof-dir> --out-dir out/traceloom
```

For local development without installation, the repository script sets
`PYTHONPATH` and uses discovery-friendly defaults:

```bash
scripts/traceloom-analyze.sh <run_dir-or-raw-msprof-dir> out/traceloom
```

## Paper Reproduction Model

The repository should not commit large profile databases. Reviewers reproduce
paper results by running the documented reference workload and profiler command
on their own multi-card Ascend/CANN machine, then running TraceLoom on the
resulting profile directory. The user-facing contract is lightweight: fill in
the device set, model path, and vLLM-Ascend checkout in
`reproduce/decode_a2a_buffer_reuse/local.env`.

Generated reproduction outputs are written under ignored `out/reproduce/` by
default:

```bash
python3 reproduce/run_reference.py decode-a2a-buffer-reuse
python3 reproduce/run_reference.py analyze-msprof /path/to/msprof_or_run_dir --name reviewer_msprof
bash run_decode_a2a_buffer_reuse.sh --env-file reproduce/decode_a2a_buffer_reuse/local.env
bash reproduce/decode_a2a_buffer_reuse/run_ab_benchmark.sh --env-file reproduce/decode_a2a_buffer_reuse/local.env
bash reproduce/decode_a2a_buffer_reuse/run_profile_pair.sh --env-file reproduce/decode_a2a_buffer_reuse/local.env
```

`run_decode_a2a_buffer_reuse.sh` is the integrated path for the local Ascend
workload: it generates TraceLoom profile configs, runs baseline and optimized
`msprof` captures, analyzes the produced msprof directories, and then builds the
comparison report from those analysis outputs.

After installation, the same entry point is available as `traceloom-reproduce`.

See [docs/reference-reproduction.md](docs/reference-reproduction.md).

## Scope

TraceLoom's core output is offline pattern discovery: readable loop structure,
CSV evidence tables, and aggregate cost summaries. Visualization is treated as
a downstream consumer of those files, not as part of the analyzer core.
